# The Sun — Product Requirements Document
*Travel Spontaneity Engine · v1.0*

---

## 1. Overview

The Sun is a personal web app designed to eliminate travel planning paralysis. It presents a small number of curated, ready-to-book weekend trip options — complete with real flight costs, weather, and things to do — so the user can stop researching and start going.

Built as a birthday gift. Scope is intentionally tight. Ship something real and delightful rather than something exhaustive.

---

## 2. The Problem

The user loves travel but rarely books trips. The cause is not lack of desire — it's analysis paralysis. Too many destinations, too many variables, too many tabs. The goal of The Sun is to remove optionality and replace it with curated momentum.

---

## 3. Goals & Non-Goals

### Goals
- Show 3–5 curated weekend trip cards immediately on load
- Each card has enough info to spark excitement, not overwhelm
- Cards incrementally increase in estimated total cost
- Clicking a card opens a full-page modal with itinerary + interactive booking checklist
- Secondary search lets user adjust budget, trip length, and vibe
- Checklist state persists via localStorage; trip details encoded in clean shareable URL

### Non-Goals (v1)
- No user accounts or authentication
- No database or server-side persistence
- No trip saving / photo logs *(v2 consideration)*
- No mobile app — responsive web only
- No real-time hotel pricing — Booking.com and Airbnb deep links only

---

## 4. User Flow

### Default Flow — Surprise Me

1. User opens the URL — no login, no setup
2. Page loads with an animated loading screen *(searching flights, checking weather, finding ideas...)*
3. 3–5 destination cards appear, ordered cheapest → most expensive for a long weekend (Fri–Mon)
4. User clicks a card → modal expands to full-page destination view
5. User works through interactive checklist, ticking off flight / hotel / Airbnb as they book
6. User can copy the URL to resume on another device — state restores from URL + localStorage

### Secondary Flow — Adjust Search

If none of the default cards appeal, the user can expand a simple search panel and adjust:

- **Budget** — e.g. £200 / £400 / £600 / custom
- **Trip length** — long weekend / 5 days / 1 week
- **Vibe** — city break / beach / culture / adventure / surprise me

> Airport is always Heathrow or Stansted — both searched simultaneously, cheapest surfaced on the card.

## 5. Destination Cards

Cards are designed like editorial travel magazine covers — they sell the feeling of a destination, not a spreadsheet of data.

| Priority | Element | Notes |
|----------|---------|-------|
| Must have | Destination name | City + Country, large and prominent |
| Must have | Hero imagery | Full-bleed background photo + 2–3 smaller images in a collage layer, sourced from Unsplash API |
| Must have | Total cost | Single estimated figure (flights + avg hotel/Airbnb combined). No breakdown on the card. |
| Must have | Weather | Averaged across trip days. Temperature + animated icon (sunny, cloudy, rain etc.) |
| Should have | Things to do | 3–4 bullet points of key experiences. Replaced by cultural event tags if something notable is happening that weekend. |
| Should have | Expand arrow | Subtle chevron — no "Book Now" button. Opens full-page modal on click. |

---

## 6. Full-Page Modal (Destination View)

Clicking a card triggers a modal that animates open to full screen. This is the immersive destination guide. Closing it returns the user to the cards view with no state lost.

### Modal Contents
- Large hero image or image collage at top
- Destination name + country
- Trip dates + total estimated cost
- Weather across each day of the trip (not just averaged)
- Itinerary overview — day-by-day suggestions generated by Claude API
- Packing suggestions — lightweight, contextual to weather and activities
- Interactive booking checklist *(see Section 7)*
- Direct links: Google Flights search, Booking.com search, Airbnb search — all pre-parameterised with destination and dates

---

## 7. Interactive Booking Checklist

The checklist is the action layer. It turns inspiration into a booking. Each item is tappable — checking it off saves state immediately.

**Default checklist items:**
- ✈ Search and book flights
- 🏨 Book accommodation (hotel or Airbnb)
- 📋 Check passport is valid
- 🛡 Arrange travel insurance
- 💳 Notify bank of travel dates
- 🎒 Pack (see packing list above)

> State persists in localStorage. URL encodes trip details so the page can be shared or resumed on another device.

---

## 8. Persistence Strategy

No database. No accounts. Two lightweight mechanisms handle all necessary state.

| Mechanism | Purpose |
|-----------|---------|
| `localStorage` | Checklist tick state. Survives tab close and return on same device. Keyed by trip ID. |
| URL encoding | Full trip details (destination, dates, cost, itinerary) encoded as a single base64 string at `/trip/[token]`. Clean, shareable, bookmarkable. Restores complete trip view on any device. |
| URL format | `thesun.app/trip/eyJkZXN0...` — single token, no query param sprawl. |

## 9. Tech Stack

| Layer | Choice | Reason |
|-------|--------|--------|
| Framework | Next.js (App Router) | React frontend + API routes in one project |
| Styling | Tailwind CSS | Utility-first, fast to build with |
| Hosting | Vercel | Single deploy, free tier sufficient for personal use |
| Backend | Next.js API routes (serverless) | Keeps all API keys server-side |
| AI | Claude API (`claude-sonnet-4-20250514`) | Destination suggestions, itinerary, packing list |
| Weather | Open-Meteo | Completely free, no API key required |
| Images | Unsplash API | Free for non-commercial use, searchable by location |
| Flights | Claude API with web search tool | Searches and synthesises live flight pricing |
| Hotels | Deep links to Booking.com + Airbnb | Good enough for v1, avoids expensive APIs |
| State | localStorage + base64 URL encoding | No database needed |

## 10. API Architecture

All external API calls go through Next.js API routes. The frontend never holds a key.

### `POST /api/search`
Accepts: `{ budget, tripLength, vibe }`

Calls Claude API to generate 3–5 destination suggestions with web search enabled for live flight pricing. Returns a structured destination array ordered by estimated cost.

### `POST /api/destination`
Accepts: `{ destination, dates }`

Calls Claude API for full itinerary + packing list. Calls Open-Meteo for day-by-day weather. Calls Unsplash for images. Returns complete destination object ready for the modal view.

### `GET /api/images?q=[destination]`
Proxies Unsplash API. Returns array of high-quality destination photos for card and modal use.

## 11. Loading Experience

The loading screen is part of the product. It should build anticipation, not feel like waiting.

- Full-screen warm background with subtle animation
- Sequential status messages that update as each step completes:
  - *"Searching flights from Heathrow and Stansted..."*
  - *"Checking the weather..."*
  - *"Finding things to do..."*
  - *"Getting your options ready..."*
- Estimated total load time: 15–25 seconds
- Subtle sun / light shimmer animation throughout

## 12. Build Order

Build iteratively. Each step is a working state you can ship or demo.

| Step | Area | What to build |
|------|------|---------------|
| 1 | Project setup | Next.js + Tailwind scaffold, Vercel deploy, env vars configured |
| 2 | API skeleton | Three API routes returning hardcoded mock data — unblock frontend dev |
| 3 | Card UI | Static destination cards with mock data — get the design right first |
| 4 | Modal UI | Full-page modal with itinerary layout and checklist component |
| 5 | Weather | Wire up Open-Meteo to `/api/destination` — real data, easy win |
| 6 | Images | Wire up Unsplash to `/api/images` — cards suddenly feel alive |
| 7 | Claude integration | Replace mock data with real Claude API calls + web search for flights |
| 8 | Persistence | localStorage for checklist + base64 URL encoding for trip sharing |
| 9 | Loading screen | Polish animated loading state with real status messages |
| 10 | Secondary search | Add search panel for budget / trip length / vibe adjustment |

## 13. API Key Security

API keys must never appear in the codebase. This is non-negotiable — a public GitHub repo with a committed key will have it scraped within minutes.

### The two-environment rule

| Environment | Where keys live |
|-------------|----------------|
| Local development | `.env.local` — git-ignored, never committed, lives only on your machine |
| Production (Vercel) | Vercel dashboard → Project Settings → Environment Variables — added manually, never touches the codebase |

### What lives in the repo

`.env.example` is the only env-related file that gets committed. It contains the key names with empty values — a template for anyone setting up the project:

```
ANTHROPIC_API_KEY=
UNSPLASH_ACCESS_KEY=
```

### Checklist before first commit
- [ ] `.env.local` is listed in `.gitignore` before any keys are added
- [ ] Run `git status` and confirm `.env.local` is not tracked
- [ ] Only `.env.example` (with empty values) is committed
- [ ] Production keys are added manually in the Vercel dashboard, not copied from any file in the repo

### Adding keys to Vercel
1. Push the project to GitHub
2. Import the repo in the Vercel dashboard
3. Before deploying, go to **Project Settings → Environment Variables**
4. Add `ANTHROPIC_API_KEY` and `UNSPLASH_ACCESS_KEY` with their real values
5. Deploy — Vercel injects them at build time, they are never exposed to the client

> Next.js API routes run server-side only. As long as keys are referenced solely within `/app/api/` routes and never imported into any client component, they will never be bundled into the frontend or exposed in the browser.
